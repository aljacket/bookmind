## Context

Today `POST /recommendations` (bookmind-server/main.py) accepts a 6-field `UserPreferences` payload (genre, bookLength, period, complexity, purpose, learningGoal, lang) and renders one of three localized templates (en/es/it) into a single GPT-4o-mini call with `max_tokens=300`. The frontend (`src/views/PreferencesPage.vue`) is a Vue 3 dropdown form. Saved recommendations, reading list, cover lookup via Google Books, IndexedDB cache, Firebase auth, and skeleton loaders all sit downstream of this call and stay untouched. The competitive analysis behind this change is in `proposal.md`. The app ships via Capacitor to web, iOS, and Android (`com.bookmind.app`); a Google Play launch is the immediate goal after this change.

## Goals / Non-Goals

**Goals:**
- Replace the form with a 2–3 turn conversation that elicits a richer signal (a loved book + current mood + one disambiguating answer) than the current dropdowns can.
- Keep the user-perceived flow under ~30 seconds and ≤3 turns — this is a guided chat, not an open chatbot.
- Preserve the existing 3-language support (en/es/it) end-to-end: opener prompt, clarifier prompt, final recommendations all in the user's language.
- Keep per-session LLM cost under ~$0.001 by staying on `gpt-4o-mini` and bounding token budgets.
- Position the AI use as transparent (one-line disclosure shown in the chat) — direct response to the Fable controversy referenced in the proposal.

**Non-Goals:**
- Free-form open-ended chat. The flow is bounded; after turn 3 the user gets recommendations or a "let's start over" option, not an unlimited dialogue.
- Memory across sessions or building a user profile. Each session is independent — this is a deliberate trust posture.
- Streaming responses, voice input, multi-modal input.
- Persisting transcripts server-side (privacy-by-default; matches "no profile built" claim).
- Anything related to the Play Store launch itself (icon, splash, signing, store listing) — covered by a follow-up change.

## Decisions

### Decision 1: Three-turn shape with one LLM-generated turn

Fixed structure: **(T1) opener** "What are you in the mood to read right now?" → **(T2) anchor** "Tell me a book you loved (or hated) recently — and what stuck with you" → **(T3) clarifier** generated by the LLM from T1+T2 (e.g. "Short read for the weekend, or something to sink into?").

**Why this over alternatives:**
- *vs. open-ended chat:* bounded UX is shippable in days, not weeks; users finish; cost is predictable. Open chat invites jailbreaks and runaway sessions.
- *vs. fixed 3rd question:* a static T3 wastes the LLM. The whole point of using GPT here is that T3 should adapt — if the user already said "I want a 200-page contemporary romance", T3 should ask about tone (cozy vs. angsty), not length.
- *vs. 2 turns:* the anchor + opener alone often leave critical ambiguity (length, tone, complexity). One adaptive question closes most of the gap without overstaying.

### Decision 2: Two backend endpoints, transcript-as-payload

- `POST /recommendations/clarify` → input: `{ lang, transcript: [{role, content}] }` of length 2 (T1 + T2). Output: `{ question: string }`.
- `POST /recommendations` → input: `{ lang, transcript: [...] }` of length ≥3 (T1 + T2 + T3). Output: existing `BookRecommendation[]` shape (`{title, author, reason}`).

**Why this over alternatives:**
- *vs. one endpoint that does both:* cleaner separation, easier caching/retry, and lets the frontend show the clarifier question with its own loading state (perceived speed).
- *vs. WebSocket / streaming:* overkill for two short request/response cycles; complicates Capacitor and the FastAPI deploy.
- *vs. keeping the old `UserPreferences` schema as a fallback:* explicit BREAKING change is cleaner for a pre-launch app with no production users. No dual-write, no migration shim.

### Decision 3: Strong system prompt + low temperature on T3, normal temperature on recommendations

T3 generation uses `temperature=0.3` (we want a focused, on-topic question, not creative tangents). Recommendation generation keeps `temperature=0.7` and `max_tokens=400` (slightly up from 300 to give room for richer reasons that cite the user's words).

Both system prompts include explicit guardrails: "Only ask/recommend about books. If the user asks about anything else, redirect to books." This addresses jailbreak risk and keeps the app on-mission.

### Decision 4: Localization stays in the prompt builder, not the route

Move the en/es/it templates out of `main.py` into `bookmind-server/prompts.py` (new) with two functions: `build_clarifier_prompt(lang, transcript)` and `build_recommendation_prompt(lang, transcript)`. Same languages preserved.

**Why:** `main.py` is already long and language-switching nested in the route is a smell. Extraction is cheap and makes adding a 4th language a one-file change.

### Decision 5: AI transparency disclosure as a static UI element, not LLM output

A single line above/below the chat: *"BookMind sends only this conversation to OpenAI. We don't build a profile and don't store your messages."* (and i18n equivalents). Hard-coded in the Vue view, not generated.

**Why:** Lifted directly from the Fable lesson — users now distrust opaque AI in book apps. A clear, non-AI-generated disclosure is more credible than any LLM-produced one. Costs nothing.

## Risks / Trade-offs

- **Risk: Users skip the clarifier and want recommendations after T2.** → Mitigation: Make T3 dismissible with a "skip" button that submits to `/recommendations` with the 2-turn transcript. The recommendation prompt must handle both lengths gracefully.

- **Risk: LLM-generated T3 is repetitive or low quality.** → Mitigation: System prompt for clarifier explicitly forbids generic questions ("don't ask about genre — they already told you") and gives 3 example dimensions to probe (length/commitment, tone/mood, familiarity vs. novelty). If it still drifts, log T3 outputs for the first ~200 sessions and tune.

- **Risk: Cost doubles per recommendation (2 calls instead of 1).** → Mitigation: Both calls are gpt-4o-mini, both bounded (clarifier ≤80 tokens out, recommendations ≤400 tokens out). Worst case ~$0.001/session vs. current ~$0.0005. Acceptable for the qualitative jump.

- **Risk: Users find chat slower than dropdowns and bounce.** → Mitigation: T1+T2 are visible immediately on page load (not gated by a previous answer); only T3 is async. Total interaction stays ≤30s. Skeleton loaders already exist and apply naturally.

- **Risk: Jailbreak / off-topic abuse via free-text input.** → Mitigation: System prompts on both calls scope strictly to books. Frontend caps each turn at 500 chars. No persistence means no poisoning across sessions.

- **Risk: Localization regressions during the en/es/it move into prompts.py.** → Mitigation: Snapshot the existing prompt strings into a test fixture before refactor; assert the new builder produces equivalent output for the equivalent inputs.

- **Trade-off: Removing the typed `UserPreferences` enums means we lose the ability to do server-side validation of "valid genre/length/etc."** This is intentional — the whole point is that a free-text "cozy mystery for a rainy Sunday" is a better signal than an enum. We accept slightly noisier inputs in exchange for richer ones.

## Migration Plan

Pre-launch app, no production users → hard cutover, no compatibility shim:

1. Land backend changes (new endpoints, prompts.py module). Old `/recommendations` request shape returns HTTP 422 by virtue of the new Pydantic model.
2. Land frontend changes in the same release: rewrite `PreferencesPage.vue` as a chat view, update `bookRecommendation.ts` service.
3. Run a manual smoke pass in all 3 languages (en/es/it) against the dev server.
4. **Rollback:** `git revert` of the merge commit. No DB state to undo, no third-party config to flip back, no users to migrate.

- **Localization of the static disclosure text** — Italian and Spanish wording should be reviewed by the user (native speaker) before shipping.
- **Should the clarifier be cacheable?** Same (T1, T2) likely produces near-identical T3. A 5-minute in-memory cache could trim cost ~30%. Defer until we see real traffic; not blocking.
