### Requirement: Conversation initiation
The system SHALL present the user with a chat-style view as the entry point to requesting a recommendation, replacing the existing dropdown form. The view SHALL display the first prompt ("opener") immediately on load, asking the user in their selected language (en/es/it) what they are in the mood to read right now.

#### Scenario: Opener is shown on page load
- **WHEN** the user navigates to the recommendation entry route
- **THEN** the chat view renders with the opener prompt visible and a free-text input focused, with no LLM request having been made yet

#### Scenario: Opener is localized
- **WHEN** the app language is set to "it"
- **THEN** the opener prompt is rendered in Italian, and the same applies for "en" and "es"

### Requirement: Anchor turn
After the user submits the opener answer, the system SHALL present a second prompt asking the user to name a book they recently loved or disliked and what stuck with them. The system SHALL NOT call the LLM between the opener answer and showing this second prompt.

#### Scenario: Anchor prompt follows opener locally
- **WHEN** the user submits a non-empty opener answer
- **THEN** the chat view appends the anchor prompt without any network request

#### Scenario: Empty opener answer is rejected
- **WHEN** the user attempts to submit an empty or whitespace-only opener answer
- **THEN** the system blocks submission and surfaces an inline validation message in the user's language

### Requirement: Dynamic clarifier turn
After the user submits the anchor answer, the system SHALL request a clarifying question from the backend (`POST /recommendations/clarify`) generated by the LLM from the prior two turns. The returned question SHALL be in the user's selected language and SHALL probe a dimension not already covered by the user's answers (such as length/commitment, tone/mood, or familiarity vs. novelty).

#### Scenario: Clarifier is generated and shown
- **WHEN** the user submits the anchor answer
- **THEN** the frontend sends `{ lang, transcript: [opener, anchor] }` to `POST /recommendations/clarify`, shows a loading state, and renders the returned `question` as the next chat turn

#### Scenario: Clarifier can be skipped
- **WHEN** the clarifier question is shown
- **THEN** a "skip" affordance is available that submits the recommendation request with the 2-turn transcript instead of waiting for a 3rd answer

#### Scenario: Clarifier stays on topic
- **WHEN** the clarifier endpoint generates a question
- **THEN** the system prompt restricts output to questions about the reader's book preferences, and any non-book question is filtered or regenerated

### Requirement: Recommendation generation
The system SHALL accept a transcript of length 2 or 3 at `POST /recommendations` and return exactly 3 book recommendations, each with a `title`, `author`, and a `reason` that cites or paraphrases at least one specific phrase from the user's transcript.

#### Scenario: Three recommendations returned
- **WHEN** the frontend posts a valid transcript with `lang` set
- **THEN** the response is a list of 3 `BookRecommendation` objects in the user's language

#### Scenario: Reason cites the user's words
- **WHEN** the recommendation prompt is built
- **THEN** the system prompt instructs the model to ground each `reason` in something the user actually said, not a generic blurb

#### Scenario: Old preferences payload is rejected
- **WHEN** a client posts the legacy `UserPreferences` shape (genre/bookLength/period/complexity/purpose/learningGoal)
- **THEN** the server responds with HTTP 422 (Pydantic validation error)

#### Scenario: Localized prompts are used
- **WHEN** `lang` is "en", "es", or "it"
- **THEN** the corresponding language template from `bookmind-server/prompts.py` is used for both the clarifier and the recommendation prompt

### Requirement: AI transparency disclosure
The chat view SHALL display a static, non-AI-generated one-line disclosure stating that the conversation is sent to OpenAI, that no user profile is built, and that messages are not stored server-side. The disclosure SHALL be localized in en/es/it.

#### Scenario: Disclosure is visible during the chat
- **WHEN** the chat view is rendered (at any turn from opener through recommendations)
- **THEN** the disclosure line is visible without requiring a click or expansion

#### Scenario: Disclosure text is hard-coded
- **WHEN** the chat view renders the disclosure
- **THEN** its text is sourced from a static i18n dictionary in the frontend, not from any LLM call

### Requirement: No server-side persistence of conversation
The backend SHALL NOT persist the conversation transcript to any datastore, log file, or third-party system beyond the in-flight LLM request. The transcript SHALL exist only for the duration of the request handler.

#### Scenario: Transcript not written to disk
- **WHEN** a recommendation request is handled
- **THEN** the handler does not write the transcript to a database, file, or persistent log; only operational metadata (timestamp, status, latency) may be logged

#### Scenario: No cross-session memory
- **WHEN** the same user starts a new conversation
- **THEN** the system has no record of any prior conversation and treats the session as wholly independent

### Requirement: Bounded turn budget
The chat SHALL be capped at 3 user turns. After the user receives recommendations, the only further actions offered are "start over" (resets to the opener) or interacting with the recommendation cards. There SHALL be no open-ended follow-up dialogue.

#### Scenario: No 4th turn is offered
- **WHEN** the recommendations are rendered
- **THEN** the chat input is disabled or hidden, and only "start over" plus existing card actions (save, mark as read, etc.) are available

#### Scenario: Per-turn input is length-capped
- **WHEN** the user types into any turn input
- **THEN** the input enforces a hard cap of 500 characters
